Introdução

No dia 29 de agosto de 2024, entrevistamos o StakeHolder Alexandre Marinho Salles, estudante de administração da FGV EBAPE, para a obtenção dos requisitos para a aplicação de máquinas de café na Fundação Getúlio Vargas.

Requisitos funcionais

  • RF 02- Fornecer comprovante digital para solucionar eventuais problemas de pagamento de maneira prática. Realizar todos os processos de pagamento internamente na própria aplicação, sem utilizar apps terceiros.
  • RF 04- Emitir um email para o usuário contendo recibo após a efetuação da compra.
  • RF 07- Implementar um sistema de avaliação no qual o usuário pode avaliar o produto ou serviço através de comentários e uma escala de avaliação variando de 0 a 5 estrelas.
  • RF 18- Notificar o usuário a respeito promoções, serviços e eventuais modificações nas máquinas.
  • RF 19- Possibilitar o usuário configurar as notificações de acordo com suas preferências.
  • RF 22- Cada usuário pode adicionar foto e outras informações pessoais no seu perfil. Cada usuário possui um histórico de compras (informação protegida) que pode ser compartilhada.
  • RF 23- Cada usuário pode seguir outros usuários da plataforma, podendo visualizar o histórico de compras uns dos outros.
  • RF 24- Os usuários podem trocar mensagens privadas entre si.
  • RF 27- A plataforma possui um sistema de gamificação.
  • RF 28- A plataforma possui um sistema de conquistas.
  • RF 29- O sistema de gamificação deve possuir níveis.
  • RF 32- É possível personalizar o aplicativo conforme o estilo do usuário.
  • RF 33- O sistema deve possuir um mecanismo para reportar problemas com produtos ou serviços.
  • SRF 07- É possível moderar comentários após os mesmos serem publicados.
  • A forma de cadastro é livre (a FGV não fornece).
  • A plataforma deve possuir um mecanismo de moeda virtual.
  • A plataforma deve possuir integração com outras redes sociais.
  • É permitida a criação de perfis privados, de modo que as informações de um usuário fiquem restritas aos seus seguidores.

Requisitos não-funcionais

  • SRF 03- O processamento e o pagamento de produtos e serviços devem ser realizados de maneira segura.
  • O sistema esperado deve possuir a possibilidade de ser integrado com outros sistemas da instituição, porém não integrado com a SRA.
  • As informações de todos os usuários devem ser protegidas.
  • Espera-se alguma medida de proteção contra ataques cibernéticos e fraudes.
  • O sistema deve possuir interface fácil e intuitiva aos usuários.
  • O sistema possui uma velocidade padrão de desempenho.
  • O aplicativo estará disponível para dispositivos móveis.
  • Disponibilidade: O sistema deve estar disponível e operacional 24 horas por dia, 7 dias por semana (24/7), garantindo que os usuários possam acessá-lo a qualquer momento, sem interrupções.

Histórias de Usuário

  • Como um usuário da faculdade, Eu quero saber quais máquinas de café estão funcionando, Para que eu possa escolher a mais próxima ou conveniente para utilizar.
  • Como um usuário da faculdade, Eu quero saber quais produtos estão disponíveis nas lojas espalhadas, Para que eu possa encontrar o que preciso de forma rápida e eficiente.
  • Como um usuário da faculdade, Eu quero saber o que meus amigos estão comprando, Para que eu possa descobrir novas recomendações e tendências entre o meu círculo social.

Casos de Uso

  • Nome: Solicitar recebimento de café
    Fluxo principal:
    1. Usuário realiza o login.
    2. Usuário seleciona o café que deseja.
    3. Usuário seleciona o local em que deseja receber o café.
    4. Usuário confirma o pedido.
    5. A plataforma procura por outro usuário que esteja disponível para realizar o serviço.
    6. Após encontrar o entregador, o sistema processa o pagamento.
    7. O sistema confirma a compra.
    8. O café escolhido é entregue ao usuário no lugar de escolha.
    Caminho Alternativo:
    1. Usuário realiza o login.
    2. Usuário seleciona o café que deseja.
    3. Usuário seleciona o local em que deseja receber o café.
    4. Usuário confirma o pedido.
    5. A plataforma procura por outro usuário que esteja disponível para realizar o serviço.
    6. A plataforma não encontra nenhum usuário disponível para a entrega.
    7. A plataforma notifica o usuário sobre a indisponibilidade e disponibiliza a opção de cancelar o pedido ou tentar novamente.
    8. Caso o usuário tente encontrar outro entregador novamente, retorna-se ao passo 5, podendo seguir para o fluxo principal ou permanecer no caminho alternativo.
    9. Caso o usuário cancele o pedido, a operação é encerrada.
  • Nome: Verificar produtos disponíveis nas lojas
    Fluxo principal:
    1. Usuário realiza o login.
    2. Usuário seleciona a aba de produtos disponíveis.
    3. Usuário filtra os produtos por categoria ou busca.
    4. Usuário filtra os produtos por categoria ou realiza uma busca por nome.
    5. O sistema indica em quais pontos de venda do prédio os produtos estão disponíveis.
    Caminho alternativo:
    1. Usuário realiza o login.
    2. Usuário seleciona a aba de produtos disponíveis.
    3. Usuário filtra os produtos por categoria ou busca.
    4. Usuário percebe que há o produto procurado se encontra em uma categoria não compatível com o produto.
    5. Usuário acessa a área de reportar problemas no aplicativo.
    6. Usuário reporta o problema.
    7. O sistema notifica que o reporte foi recebido com sucesso.
  • Nome: Usuário realiza entrega
    Fluxo principal:
    1. Usuário realiza login.
    2. Usuário analisa as opções de entrega disponível.
    3. O sistema verifica os serviços disponíveis.
    4. Usuário seleciona a entrega que deseja.
    5. O pareamento com o cliente a receber o serviço é confirmado.
    6. O café é entregue no local estabelecido.
    7. O entregador recebe sua recompensa em moedas virtuais referente ao serviço prestado.
    Caminho alternativo:
    1. Usuário realiza login.
    2. Usuário analisa as opções de entrega disponível.
    3. O sistema verifica os serviços disponíveis.
    4. O sistema informa que não há opções de serviços disponíveis.
    5. A plataforma notifica o entregador sobre a baixa demanda de usuários e disponibiliza a opção de esperar por possíveis clientes ou cancelar a procura.
    6. Caso o entregador tente encontrar outro usuário novamente, retorna-se ao passo 3, podendo seguir para o fluxo principal ou permanecer no caminho alternativo.
    7. Caso o usuário cancele a procura, a operação é encerrada.
